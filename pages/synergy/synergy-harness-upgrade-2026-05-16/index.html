Synergy Infrastructure Upgrade — NLAH + Meta-Harness Architecture

Date: 2026-05-16
Author: Mariano
Status: Complete — deployed to both_os main branch

What This Is

A structural upgrade to the Synergy operating system — the set of agents, scripts, and pipelines that run across all BOTH ventures. The upgrade makes the system more reliable, easier to debug, and easier to extend without breaking what already works.

Two research papers drove the design:

  • NLAH (arxiv:2603.25723): Every automated pipeline should be a portable, self-describing spec — explicit about what it reads, what it writes, who approves it, and what to do when it fails. Not scattered across code and prose.
  • Meta-Harness (arxiv:2603.28052): Full execution traces enable causal debugging — knowing why something routed incorrectly is 47% more accurate when you have a structured log than when you only have a success/failure summary. Registry SHA logging. Environment bootstrap before pipeline work.

The tldr: when a task ends up in the wrong Asana section, or a Slack notification doesn't arrive, we now have a single source of truth to check and a structured trace to read — instead of hunting through code.

What Changed

1 — Venture Registry (config/ventures.json)

Before: Venture slugs, Asana section GIDs, Slack IDs, and team mappings were hardcoded in three separate places — the Cloudflare Worker, asana-push.py, and slack-notify.py. Changing a venture name or adding a new one meant editing code in multiple files and redeploying.

After: One file at config/ventures.json is the single source of truth. All routing — which Asana section a task lands in, which Slack channel gets notified, which champion gets a DM — comes from this file. The Worker fetches it at runtime (no redeploy needed). The scripts load it on startup.

To add a new venture: Edit config/ventures.json. That's it.

2 — Pipeline Harness (harnesses/tuesday-sync.md)

Before: The Tuesday Sync pipeline existed — Read.ai → Worker → Asana → Slack — but it was only described in CLAUDE.md prose. No single document answered: what happens at each stage? what does failure look like? who is responsible for what? where do I check if something went wrong?

After: harnesses/tuesday-sync.md is the authoritative spec for the Tuesday pipeline. Six stages, each with:

  • Who runs it
  • What it reads and writes
  • What structured data it logs (trace schema)
  • What failure looks like and what to do

When something in the Tuesday pipeline goes wrong, this is the first file to open — before reading code.

3 — Meeting Miner Approval Contract (.claude/agents/meeting-miner.harness.md)

Before: The approval logic — when the Worker auto-approves vs. creates a pending file vs. escalates to Mariano — was implied, not written.

After: Three explicit rules:

RuleConditionAction
A — Auto-approve HMAC-verified + non-empty transcript + venture slug detected + registry loaded Items go directly to vault + Asana
B — Pending review both-secret, WhatsApp, or manual source vault/pending/ file created for champion review
C — Escalate Registry fail, auth fail, Claude fail, vault write fail Mariano gets a Slack alert

Every pipeline run logs four fields: source (how the transcript arrived), rule_applied (which rule matched), confidence (signal quality), override_by (who could intervene). This is the causal audit trail — when something routes wrong, these four fields tell you exactly why.

4 — Agent Contracts (all .claude/agents/*.md files)

Before: Agent files described what an agent does but not what it reads, what it writes, who approves it, or what failure looks like. You had to read the full file to understand the boundaries.

After: Four priority agents now have an explicit ## Contract block at the top:

Priority agents upgraded: meeting-miner, venture-update, area-update, weekly-reporter.

All other agent files have a <!-- CONTRACT: pending --> marker — they'll get full contracts when next revised.

5 — CLAUDE.md Harness Architecture Section

A new ## Harness Architecture section in CLAUDE.md explains:

  • Why these two papers were applied
  • Where the registry lives and how to use it
  • Where harness specs live
  • How to add a venture (4-step process, no code changes)
  • Where execution traces go (Supabase pipeline_runs — pending table creation)

The duplicate routing data that was in CLAUDE.md prose (venture slugs, Slack IDs, Asana GIDs) now points to config/ventures.json instead of repeating it.
