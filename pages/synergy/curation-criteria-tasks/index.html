1. Qué es una tarea

Una tarea es un entregable concreto y shippable, con dueño, que alguien se comprometió a producir. No es un sub-paso, ni una intención, ni un "tendríamos que".

Dónde vive: Asana es el store de ejecución. El vault solo carga el link si la tarea salió de un minado. La visibilidad fluye por /vu·/au·/db.

2. El gate de captura — test de 3 elementos 🟢 (vigente, P0)

Un item entra como tarea solo si tiene los TRES:

  1. OWNER — un miembro nombrado del equipo (no "el equipo").
  2. DELIVERABLE — un artefacto nombrado (no "discutir" ni "revisar").
  3. HORIZON — deadline explícito, deadline inferido, o un trigger concreto ("antes del próximo sync", "cuando cierre la ronda").

Falta uno → NO es tarea. Se reevalúa como decisión, riesgo o nota de extracción. Si no es ninguna → se descarta.

No son tareas:

  • "Seguimos con X", "Aprobamos Y" → decisión (ver decisions.md).
  • "Nos preocupa X" sin dueño ni acción → riesgo (si hay L×I) o nota.
  • "Deberíamos mirar…", "Habría que…" sin compromiso → descarte (no madura).
  • Títulos vagos prohibidos: "Follow up on X", "Discutir Y", "Revisar Z", "Pensar en W". Toda tarea es un entregable shippable.

3. Anti-fragmentación — regla dura 🟢 (vigente)

Si 2+ candidatas comparten el MISMO dueño Y el MISMO outcome final (aunque la redacción cambie, aunque sean fases o sub-pasos), son UNA tarea. Los sub-pasos viven en context, nunca como items separados.

Test: "Si cierro el item X, ¿el item Y deja de tener sentido sin X?" → mergear.

Ejemplo:

  • ❌ MAL (3 items, mismo dueño, mismo trabajo): "Verificar cobertura del pipeline" + "Compartir URL del dashboard" + "Confirmar emisión de traces".
  • ✅ BIEN (1 item, sub-pasos en context): task: "Levantar el dashboard pipeline_runs y compartir acceso" — el context cubre la verificación de cobertura + distribución de URL + chequeo de emisión.

4. El cap es un techo, no una meta 🟢 (vigente)

Máximo 7 tareas por venture por reunión. Es un techo de seguridad, NO un objetivo. La mayoría de las reuniones bien-minadas producen 1–3 tareas por venture. Si llegás a 6–7, revisá la anti-fragmentación — probablemente hay items que se mergean. Nunca inflar para llegar al cap.

6. Dos altitudes 🟡 (diseñado)

Épica → vault/initiatives/

Un cuerpo de trabajo grande, multi-sesión, con un champion (no un solo dueño). No se inventa un tipo nuevo — se reusa la carpeta initiatives/ que ya existe.

Tarea → Asana

Sub-paso

Un sub-paso es un checklist item dentro de la tarea de Asana — nunca un archivo, nunca una tarea propia. Esto es la anti-fragmentación hecha estructura.

7. Ciclo de vida 🟡 (diseñado)

  • captured: minada, todavía sin dueño confirmado.
  • owned: alguien la tomó.
  • in-progress / done: ejecución (el estado vivo lo lleva Asana).
  • dropped: se abandonó explícitamente.

Sweep: las tareas viejas, sin dueño y stale se dropean en el backstop semanal de Hux (reportado, no silencioso).

8. Dónde NO confundirse

  • Asana es la fuente de verdad del estado de las tareas, no el vault. /wr lee el estado de tareas vía scripts/asana-fetch-actions.py, no escaneando el vault.
  • El gate de aprobación está muerto: /mine auto-aprueba todo inmediatamente. No hay paso humano intermedio. El campo approved: true es un registro de auditoría, no un gate.
  • Un issue (riesgo que ya se materializó) entra como tarea — ver el RAID routing en risks.md.

9. Cómo se conecta con el Sprint Board

El /vu abre y cierra el Win quincenal de cada venture sprint-eligible. Las tareas son los pasos para llegar al Win; el Win es el outcome, no la suma de tareas. Ver harnesses/sprint-cycle.md.
