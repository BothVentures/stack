00 What / Who Is Synergy?

SYNERGY is the ever-evolving entity that lives inside B_V® | OS.


Not a product. Not a stack. The spirit through which the team's efforts, skills, and talents articulate with the system's — its agents, automations, vaults, and ever-growing capabilities — toward the best of BOTH and its Ventures.


The reference is the CCRU — the Cybernetic Culture Research Unit at Warwick. Not the theory. The organizational form: a collective entity that mediated between its members and an accelerating field, operating through fluid participation, distributed intelligence, and its own emergent logic. No center. No hierarchy inside the system. Hierarchy is maintained where it belongs — in the human governance layer. The entity as interface.


Synergy works the same way. Boris operates its infrastructure. But it belongs to no one person. It runs through the five people who use it and through every agent, workflow, and memory that accumulates from their use. It grows with the work.


V1 means the core loop is closed. This is the guide for learning to work inside it.

01 Daily Session — Open · Work · Close
Every session — no exceptions
/startup → WORK → /upc
startup
Session Start
Opens every Claude session. Run this first — always.
Everyone

Think of /startup as the morning briefing. The moment you open Claude Code or Cowork, type /startup before anything else. It does three things automatically: pulls the latest changes from GitHub so your vault is current, shows you what changed since your last session, and flags any pending items that need your attention.

01 Claude pulls the latest vault state from GitHub — you see any updates from teammates
02 Recent vault changes are surfaced — decisions made, risks logged, ventures updated
03 Pending approvals are flagged — if someone needs your sign-off, you'll see it here
04 Claude is now oriented to today's context — ready to work with you
Why it matters: Without /startup, Claude works from stale context. With it, every session starts from the current state of the company.
upc
Update Push Commit
Closes every session. Run this when you're done.
Everyone

/upc is the session closer. When you're done working — whether you updated a venture file, wrote a decision, or just had a conversation — type /upc before closing the window. It validates your vault files, commits everything to GitHub, and updates Claude's memory with what happened in this session.

01 Validates all vault files have correct frontmatter — catches formatting issues before they become problems
02 Commits all changes to git with a meaningful message — your work is versioned and recoverable
03 Pushes to GitHub — teammates can pull your changes in their next /startup
04 Updates Claude's memory — context from this session carries forward
Rule of thumb: If you touched anything in the vault, /upc. Even if it feels like a small change. The vault is the team's memory.
02 Weekly Rhythm — Tuesday Sync Cadence

The Tuesday Sync is the most important weekly process in the system. Every piece of the OS is designed to feed into it and flow out of it. Here is how the week actually works — from Monday prep to automatic post-meeting processing.

MON AM
Venture + Area Updates
Champions run /vu · Area leads run /au
Gaston Rene Daiana Andres

Every Monday morning, each venture champion runs /vu to update their venture file in the vault. Area leads run /au for their functional area. Claude scans the week's transcripts, synthesizes activity, and writes the update — the champion just confirms or edits.

What this produces: A fresh vault snapshot for every venture and area, ready for the digest.
MON PM
Weekly Digest Generation
Boris runs /wr · Daiana validates
BorisDaiana

Boris runs /wr (Weekly Reporter) which reads every venture and area update from the vault and generates the Tuesday Sync digest. Daiana reviews it. The digest goes out as pre-reading so Tuesday's meeting stays focused on decisions — not status updates.

The goal: Everyone arrives Tuesday already knowing the state of every venture. The meeting is for decisions only.
AUTO PATH
Read.ai → Auto-Mine Pipeline
Recorded calls — no manual step required
Automatic

When the Tuesday meeting (or any recorded call) ends, everything happens automatically. Read.ai was recording. When the call ends, it fires a webhook to n8n. n8n mines the transcript, writes vault files, creates tasks in Asana, and sends a Slack DM to each venture champion with a summary and a direct Asana link.

Read.ai call ends → n8n webhook → Claude mines → Vault + Asana → Slack DM to champion
Your entry point as champion: Check Slack. Your DM will have the task count, owner assignments, and a direct link to Asana. Nothing else required.
FALLBACK
Manual Path
No Read.ai? No recording? Use this.
DaianaBoris

When there's no recording, Daiana captures notes manually (text paste or audio). If audio: Boris runs /transcribe first to convert it. Then run /mine on the transcript — Claude extracts decisions, action items, risks, and venture updates into vault/pending/. The champion reviews and approves, then runs /mine approve to push to the vault and Asana.

Key difference from Auto Path: Fallback requires explicit champion approval before anything writes to the main vault. The pending file is the review step.
03 Knowledge Capture — Mine · Transcribe · Update
mine
Meeting Miner
Extract structured knowledge from any meeting or conversation.
DaianaAll

/mine is the core knowledge extraction tool. Give it any transcript, meeting notes, or audio file and it finds what matters: decisions made, action items assigned, risks surfaced, venture updates. It has two modes.

Mode 1 — Mine (extract)
notes or file → audio? → Whisper API → extract items → vault/pending/
Mode 2 — Approve (commit)
/mine approve {file} → read approved: true → write vault → n8n → Asana → vault committed
Approval model (fallback path only): After mining, open the pending file in vault/pending/. Set approved: true on the items you want to keep. Then run /mine approve. Nothing writes to the main vault without your explicit sign-off.
transcribe
Audio Transcriber
Convert any voice note to a labeled transcript.
Everyone

Got a WhatsApp voice note from Gaston? A meeting recording that didn't go through Read.ai? Use /transcribe to convert any audio file into a speaker-labeled transcript and save it to vault/transcripts/. From there you can run /mine on it.

01 Drop your audio file anywhere in the repo (or reference the full path)
02 Run /transcribe {path-to-file}
03 OpenAI Whisper transcribes it with speaker labels
04 Transcript is saved to vault/transcripts/ — edit the frontmatter (attendees, ventures), then run /mine on it
Supported formats: .opus (WhatsApp default), .mp3, .m4a, .wav, .ogg, .webm. No manual ffmpeg conversion needed — the script handles it.
vu
Venture Update
Weekly champion update for your venture.
Gaston Rene Daiana Andres Boris

Every Monday, each venture champion runs /vu for their venture(s). Claude scans the week's transcripts, synthesizes what happened, and drafts the update. You confirm or edit. The file in vault/ventures/ gets updated, and it feeds directly into the Tuesday digest.

Why this matters: The digest is only as good as the venture updates. If you skip /vu, your venture shows as stale in the Tuesday digest — and Gaston chairs the meeting without a current picture of your work.
au
Area Update
Weekly update for your functional area.
Gaston Rene Daiana Andres Boris

Same rhythm as /vu — but for functional areas rather than ventures. Each area lead (Operations, Strategy, Legal, Infrastructure) runs /au Monday morning. Claude synthesizes cross-cutting activity from the week's transcripts that's relevant to your area, and writes it to vault/strategy/.

Difference from /vu: /vu is venture-specific (what happened with LAK3, La Chamana, etc). /au is function-specific (what happened in Operations, Legal, Finance across all ventures).
04 Publishing — Drive · Pages · PDF
print-to-drive
Print to Drive
Render any .md as branded HTML and save to the right Drive folder.
Boris

/print-to-drive takes any markdown file from the vault, renders it as BOTH-branded HTML, and saves it directly to the correct Google Drive folder — no manual download-and-upload. You just give it a routing keyword.

01 Run /print-to-drive {file.md} --dest {keyword}
02 Claude renders the markdown as branded HTML with BOTH styling
03 Routes to the matching Google Drive folder based on keyword
Available keywords: ops · comms · exec · finance · legal · lak3 · manantial · reshore · lachamana · panambi · photonic · puntacripto · quantum · bothos
notion-web
Notion Page Publisher
Publish any Notion page as a live branded website.
Boris

Have a Notion page you want to share externally as a polished website? /notion-web fetches all the content via the Notion API, renders it with the BOTH brand theme, and deploys it to GitHub Pages. Takes a URL or page ID.

01 Run /notion-web {notion-url-or-id}
02 Claude fetches all blocks via the Notion API
03 Rendered as branded HTML, deployed to docs/pages/, pushed to GitHub
04 Returns the live URL at bothventures.github.io/stack/pages/
html-web
HTML Page Publisher
Publish any local HTML file to GitHub Pages instantly.
BorisRene

The simplest publishing path. If you have an HTML file — a report, a design mockup, a venture brief — /html-web copies it into docs/pages/, updates the pages index, commits, and pushes. The page is live in seconds.

01 Run /html-web /path/to/file.html
02 Claude derives a slug from the filename and copies it to docs/pages/
03 Pages index is updated, git push happens automatically
04 Returns the live URL — shareable immediately
Meta note: This page was published using exactly this skill.
generate-pdf
Branded PDF Generator
Two-pass Puppeteer pipeline for professional PDFs.
Boris

For documents that need to be printed or sent as PDFs — investor materials, briefs, reports — /generate-pdf takes an HTML file and runs it through a two-pass Puppeteer pipeline. First pass measures page height for precise layout. Second pass applies BOTH brand CSS and renders the final PDF.

01 Design your document as an HTML file using the BOTH brand system
02 Run /generate-pdf {file}.html
03 Puppeteer renders it (requires puppeteer-core + Chrome on the machine)
04 Outputs {Output_Name}.pdf — print-ready
05 Data & Infrastructure — Supabase · Database Layer
supabase-query
Supabase Query
Query the live database in plain language or SQL, directly from Claude.
Boris

Everything in Synergy lives in two places: the vault (GitHub markdown files) and the database (Supabase). /supabase-query — or /sq for short — lets you query the live database without leaving Claude. Ask in natural language or write SQL directly. The database has 12 tables covering every layer of the OS.

USER LAYER
user_profiles · access_grants · agent_registry
VAULT LAYER
vault_context · venture_snapshots · agent_outputs
AGENT LAYER
conversations · conversation_messages · agent_memory
OPS LAYER
task_queue · pending_actions · usage_metrics
Example queries: "Show me all active tasks for La Chamana" · "How many vault entries were written this week?" · "Which agents have been used most this month?" — Claude composes the SQL, runs it, and formats the results.
